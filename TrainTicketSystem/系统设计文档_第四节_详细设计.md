### 4.1.2 视图设计

**v_daily_financial_report（每日财务报表视图）**

```sql
CREATE OR REPLACE VIEW v_daily_financial_report AS
SELECT
    t.train_id,
    CURDATE() AS sale_date,
    COUNT(o.order_id)                                                   AS total_orders,
    SUM(CASE WHEN o.status = 1 THEN 1 ELSE 0 END)                       AS paid_orders,
    SUM(CASE WHEN o.status = 2 THEN 1 ELSE 0 END)                       AS refunded_orders,
    COALESCE(SUM(o.price), 0)                                           AS total_sales,
    COALESCE(SUM(CASE WHEN o.status = 2 THEN o.price ELSE 0 END), 0)    AS refund_amount,
    COALESCE(SUM(CASE WHEN o.status = 1 THEN o.price ELSE 0 END), 0)    AS net_revenue
FROM train_info t
LEFT JOIN orders o ON t.train_id = o.train_id
GROUP BY t.train_id;
```

**核算恒等式**：`net_revenue = total_sales - refund_amount`

**设计要点**：
- `total_sales` 统计所有订单金额（含已退票），反映真实营收轨迹
- `refund_amount` 仅统计已退票订单
- `net_revenue` 仅统计当前有效（已支付）订单，为实际留存收入

### 4.1.3 触发器设计

#### 触发器 1：trg_after_order_insert（购票自动记账）

```sql
CREATE TRIGGER IF NOT EXISTS trg_after_order_insert
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    IF NEW.status = 1 THEN
        INSERT INTO transaction_logs (order_id, user_id, action_type, amount, snapshot_bitmap)
        VALUES (NEW.order_id, NEW.user_id, 'PAY', NEW.price, NEW.buy_mask);
    END IF;
END;
```

**触发时机**：向 `orders` 表插入新订单（status=1，已支付状态）时  
**执行动作**：自动向 `transaction_logs` 写入一条 `PAY`（支付入账）流水  
**设计目的**：购票记账无需应用层手动 INSERT，由数据库底层保障日志完整性

#### 触发器 2：trg_after_order_update（退票自动记账）

```sql
CREATE TRIGGER IF NOT EXISTS trg_after_order_update
AFTER UPDATE ON orders
FOR EACH ROW
BEGIN
    IF OLD.status = 1 AND NEW.status = 2 THEN
        INSERT INTO transaction_logs (order_id, user_id, action_type, amount, snapshot_bitmap)
        VALUES (NEW.order_id, NEW.user_id, 'REFUND', NEW.price, NEW.buy_mask);
    END IF;
END;
```

**触发时机**：`orders` 表状态从 1（已支付）更新为 2（已退票）时  
**执行动作**：自动向 `transaction_logs` 写入一条 `REFUND`（退款支出）流水  
**边界保护**：仅在 `OLD.status=1 AND NEW.status=2` 时触发，避免重复退票或无效状态变更产生错误日志

### 4.1.4 位图座位管理模型

**核心原理**：一个车次有 N 个经停站 → 有 N-1 个运行区间。每个区间的占用状态用 1 个 bit 表示。

```
G101 经停站：北京南(0) → 济南西(1) → 南京南(2) → 上海虹桥(3)
区间分布：   [bit0]        [bit1]        [bit2]
```

**位图运算**：

| 操作 | 公式 | 示例 |
|---|---|---|
| 购票置位 | `seat_bitmap = seat_bitmap | buy_mask` | 0 | 0b111 = 7 |
| 退票清零 | `seat_bitmap = seat_bitmap ^ buy_mask` | 7 ^ 7 = 0 |
| 冲突检测 | `(seat_bitmap & buy_mask) != 0` | (7 & 0b001) = 1 → 冲突 |

**掩码计算公式**：

```python
segment_count = end_stop_index - start_stop_index
buy_mask = ((1 << segment_count) - 1) << start_stop_index

# 示例：北京南(stop=0) → 上海虹桥(stop=3)
# segment_count = 3, buy_mask = ((1<<3)-1)<<0 = 0b111 = 7
```

**冲突检测示例**：

```
座位 01A 当前 seat_bitmap = 0b010（济南西→南京南已售）
用户想买 北京南→上海虹桥（buy_mask = 0b111）
碰撞检测: 0b010 & 0b111 = 0b010 ≠ 0 → 冲突，拒绝购买
```

## 4.2 API 接口详细设计

系统共 **17 个 API 路由**，统一返回 JSON 格式，异常返回 `{"error": "..."}` + HTTP 状态码。

### 4.2.1 认证类接口

#### POST /api/register（用户注册）

**请求体**：`{"username": "zhangsan", "password": "123456"}`

**处理流程**：
1. 校验用户名 ≥ 2 字符、密码 ≥ 4 字符
2. `SELECT 1 FROM users WHERE username = %s` 查重
3. 调用 `hash_password(password)` 生成 PBKDF2-SHA256 哈希（格式 `salt$hex_digest`）
4. 生成 `user_id = 'U_' + uuid[:12]`
5. `INSERT INTO users (user_id, username, password, role_type) VALUES (..., 'USER')`

**成功响应**：`{"success": true, "message": "注册成功，请登录", "user_id": "U_XXXX", "username": "zhangsan"}`

**失败响应**：409 用户名已注册 / 400 参数不合法 / 500 数据库异常

#### POST /api/login（用户登录）

**请求体**：`{"username": "admin", "password": "123456"}`

**处理流程**：
1. 调用 `lookup_user(username, password)` 查询并验密
2. 写 Session：`session['role']`、`session['role_type']`、`session['user_id']`、`session['username']`
3. 返回 `{"success": true, "role": "ADMIN", "username": "admin"}`

**密码验证流程**：`users` 表查用户 → `verify_password(明文, 哈希值)` → 若失败回退内置硬编码种子账户

#### POST /api/logout

**处理**：`session.clear()`，前端同步清空 localStorage

### 4.2.2 用户业务接口

#### GET /api/trains（车次列表）

**鉴权**：无  
**SQL**：

```sql
SELECT t.train_id, t.train_no, t.departure, t.arrival,
       t.departure_time, t.total_seats,
       COALESCE(NULLIF(t.price, 0), tm.base_rate, 100.00) AS base_rate
FROM train_info t
LEFT JOIN train_models tm ON tm.type_code = LEFT(t.train_id, 1)
ORDER BY t.train_id
```

**返回**：`[{train_id, train_no, departure, arrival, departure_time, total_seats, base_rate}, ...]`

#### GET /api/stops/<train_id>（经停站查询）

**鉴权**：无  
**SQL**：`SELECT * FROM train_stops WHERE train_id = %s ORDER BY stop_index`  
**返回**：`[{train_id, station_name, stop_index, arrival_time, departure_time}, ...]`

#### GET /api/seats（座位状态查询）

**鉴权**：无  
**参数**：`?train_id=G101`  
**SQL**：联查 seat_status ⨝ seat_classes ⨝ train_info ⨝ train_models  
**返回**：含 seat_id、seat_no、class_code、seat_bitmap、class_name、price_multiplier、base_rate

**前端渲染逻辑**：
1. 根据当前选择的上下车站计算 `currentMask`
2. 对每个座位执行 `(seat_bitmap & currentMask) != 0` 判定占用/空闲
3. 空闲 → 绿色可点击；占用 → 红色禁用

#### GET /api/query_routes（智能路线检索）

**鉴权**：无  
**参数**：`?from_station=北京&to_station=上海`

**模式 A（有参数）— LIKE 模糊匹配**：

```sql
SELECT ... FROM train_info t
JOIN train_stops s_start ON t.train_id = s_start.train_id
JOIN train_stops s_end   ON t.train_id = s_end.train_id
LEFT JOIN train_models m ON m.type_code = LEFT(t.train_id, 1)
WHERE s_start.station_name LIKE '%北京%'
  AND s_end.station_name   LIKE '%上海%'
  AND s_start.stop_index < s_end.stop_index
```

**模式 B（无参数）— 浏览全部**：取每个车次的首站（stop_index=0）和末站（MAX stop_index）

**定价熔断**：`COALESCE(NULLIF(t.price, 0), m.base_rate, 100.00) AS base_rate`  
→ `t.price > 0` 使用管理员自定义价，`=0` 走车型字典，都没有则兜底 100

#### GET /api/get_train_schedule（完整时刻表）

**鉴权**：无  
**参数**：`?train_id=G101`  
**SQL**：`SELECT stop_index, station_name, arrival_time, departure_time FROM train_stops WHERE train_id = %s ORDER BY stop_index`  
**前端渲染**：垂直时间轴 CSS，首站 🟢 绿色实心圆点、末站 🔴 红色实心圆点、中间站 ○ 白底蓝边圆点

#### POST /api/buy（购票）

**鉴权**：登录态  
**请求体**：`{"train_id": "G101", "seat_no": "01A", "start_station": "北京南", "end_station": "上海虹桥"}`

**事务流程（7 步原子操作）**：

```
conn.begin()
  ├─ Step 1: SELECT ... FOR UPDATE 锁定座位行（获取 seat_id, bit_map, class_code）
  ├─ Step 2: Python 层冲突检测 (current_bitmap & buy_mask) != 0 → rollback
  ├─ Step 3: UPDATE seat_bitmap = bitmap | buy_mask（OR 置位扣减库存）
  ├─ Step 4: 服务端重新计价
  │    ├─ COALESCE(NULLIF(ti.price,0), tm.base_rate, 100) → base_rate
  │    ├─ seat_classes.price_multiplier → multiplier
  │    └─ Decimal(segs) × Decimal(base_rate) × Decimal(multiplier) → actual_price
  ├─ Step 5: INSERT INTO orders (order_id, ..., actual_price, 'PAID', 1)
  ├─ Step 6: 触发器 trg_after_order_insert 自动写入 transaction_logs
  └─ Step 7: COMMIT（任一步失败 → ROLLBACK 全部）
```

**安全设计**：
- `FOR UPDATE` 悲观锁杜绝并发超卖
- `actual_price` 由后端 Decimal 高精度重算，不信任前端任何金额
- 触发器保证日志写入与订单状态变更强一致

#### POST /api/refund（退票）

**鉴权**：登录态 + 权限校验（ADMIN 可退任意单 / USER 只能退自己的）  
**请求体**：`{"order_id": "ORDXXXX"}`

**事务流程（6 步原子操作）**：

```
conn.begin()
  ├─ Step 1: SELECT user_id, seat_id, buy_mask FROM orders
  │          WHERE order_id = %s AND order_status = 'PAID' FOR UPDATE
  ├─ Step 2: 权限校验（ADMIN 全权 / USER 校验 owner_id）
  ├─ Step 3: UPDATE orders SET order_status='REFUNDED', status=2
  │          WHERE order_id = %s AND order_status='PAID'（rowcount 强校验）
  ├─ Step 4: UPDATE seat_status SET seat_bitmap = bitmap ^ buy_mask（XOR 释放）
  │          WHERE seat_id = %s（rowcount 强校验）
  ├─ Step 5: 触发器 trg_after_order_update 自动写入 transaction_logs
  └─ Step 6: COMMIT（任一步 rowcount≠1 → ROLLBACK）
```

#### GET /api/my_orders（我的订单）

**鉴权**：登录态  
**核心算法**：位运算反向解码真实行程

```python
# 从 buy_mask 反推上下车站
start_index = (buy_mask & -buy_mask).bit_length() - 1  # 最右 1 的位置
end_index   = buy_mask.bit_length()                      # 最左 1 的下一位
# 示例：buy_mask=7(0b111) → start=0, end=3 → 北京南 → 上海虹桥
```

### 4.3.3 权限隔离实现

## 4.4 安全设计

### 4.4.1 密码安全

- 存储格式：PBKDF2-SHA256，100,000 轮迭代，随机 16 字节盐，格式 `salt$hex_digest`
- 明文密码从不落库，从不日志输出
- 登录接口不返回密码哈希

### 4.4.2 并发安全（防超卖）

**三层防护**：

| 层 | 机制 | 作用 |
|---|---|---|
| 数据库层 | `SELECT ... FOR UPDATE` 悲观行锁 | 同一座位同时只能一个事务操作 |
| 应用层 | Python `(bitmap & mask) != 0` 内存检测 | 二次确认，拦截锁内并发冲突 |
| 位图层 | OR 置位 / XOR 清零 | 位运算原子性，不会出现中间态 |

### 4.4.3 事务安全

以下操作全部包裹在 `conn.begin()` ... `conn.commit()` 中，任一步失败触发 `conn.rollback()`：

- 购票（购票 7 步）
- 退票（退票 6 步）
- 发布新车次（3 类 INSERT 事务）
- 删除车次（4 类 DELETE 事务）

### 4.4.4 数据完整性

- 订单状态变更：`rowcount` 强校验（必须 == 1），防止重复退票
- 座位释放：`rowcount` 强校验，防止误操作
- 外键约束：`train_stops → train_info` 设置 `ON DELETE CASCADE`
- 触发器防篡改：`transaction_logs` 由数据库底层触发器写入，应用层无法跳过

### 4.4.5 定价安全

```
全系统 base_rate 统一取值公式（5 处一致）：
COALESCE(NULLIF(train_info.price, 0), train_models.base_rate, 100.00)

优先级：管理员自定义价 > 车型字典价 > 兜底 100
```

购票时 `actual_price = Decimal(segments) × Decimal(base_rate) × Decimal(multiplier).quantize('0.00')`，前端传来的金额完全忽略。

| `/api/register`     | POST | 注册新用户     | `INSERT INTO users (... role_type='USER')`                   |
| `/api/trains`       | GET  | 车次下拉列表   | `SELECT ... FROM train_info LEFT JOIN train_models`          |
| `/api/stops/<id>`   | GET  | 经停站查询     | `SELECT ... FROM train_stops ORDER BY stop_index`            |
| `/api/seats`        | GET  | 座位状态矩阵   | `seat_status ⨝ seat_classes ⨝ train_info ⨝ train_models`     |
| `/api/query_routes` | GET  | 智能站到站检索 | `train_info ⨝ train_stops×2 ⨝ train_models`，LIKE 模糊 / 全量双模式 |

### 登录接口（4 个）

| 路由             | 方法 | 鉴权    | 用途                                         |
| ---------------- | ---- | ------- | -------------------------------------------- |
| `/api/logout`    | POST | session | 清空会话                                     |
| `/api/buy`       | POST | session | 购票（事务：锁→检测→扣库存→计价→写订单）     |
| `/api/refund`    | POST | session | 退票（事务：锁→权限→改状态→XOR 释放→记日志） |
| `/api/my_orders` | GET  | session | 我的订单（位运算反向解码真实行程）           |

### 管理接口（5 个）

| 路由                     | 方法   | 用途                                                         |
| ------------------------ | ------ | ------------------------------------------------------------ |
| `/api/admin/trains`      | GET    | 管理员车次列表                                               |
| `/api/admin/trains/<id>` | DELETE | 级联删除车次（orders→seat_status→train_info，stops CASCADE） |

| `/api/finance/logs`      | GET    | 流水明细（读 `transaction_logs`，倒序 LIMIT 100）            |
