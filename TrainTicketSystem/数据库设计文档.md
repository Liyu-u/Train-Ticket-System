## 1. train_info — 列车车次信息表

存储所有可售票车次的基础信息。

### DDL

```sql
CREATE TABLE train_info (
    train_id       VARCHAR(20)   PRIMARY KEY   COMMENT '车次编号，如 G101',
    train_no       VARCHAR(20)                 COMMENT '车次号',
    departure      VARCHAR(100)  NOT NULL      COMMENT '始发站',
    arrival        VARCHAR(100)  NOT NULL      COMMENT '终点站',
    departure_time DATETIME      NOT NULL      COMMENT '发车时间',
    price          DECIMAL(10,2)               COMMENT '票价（元）',
    total_seats    INT                         COMMENT '总座位数'
) ENGINE=InnoDB COMMENT='列车基础信息表';
```

> **注意**：`01_init_db.sql` 中列名为 `start_station` / `end_station`，实际运行时代码使用 `departure` / `arrival`。`db_setup.py` 和 `webapp.py` 均按后者的列名读写，`webapp.py` 在查询时通过 `IFNULL` 做了兼容处理。

### 字段说明

| 列名 | 类型 | 键 | 非空 | 默认值 | 说明 |
|------|------|:--:|:----:|--------|------|
| `train_id` | VARCHAR(20) | PK | ✅ | — | 车次编号，如 `G101` |
| `train_no` | VARCHAR(20) | | | NULL | 车次号（db_setup.py 动态添加列） |
| `departure` | VARCHAR(100) | | ✅ | — | 始发站名称 |
| `arrival` | VARCHAR(100) | | ✅ | — | 终点站名称 |
| `departure_time` | DATETIME | | ✅ | — | 计划发车时间 |
| `price` | DECIMAL(10,2) | | | NULL | 单程票价（db_setup.py 动态添加列） |
| `total_seats` | INT | | | NULL | 该车次可售票的座位总数 |

### 示例数据

## 2. seat_status — 座位区间状态表（核心）

这是整个系统最核心的表，使用**二进制位图**记录每个座位在各行驶区间的占用情况。

### DDL

```sql
CREATE TABLE seat_status (
    seat_id       INT AUTO_INCREMENT PRIMARY KEY  COMMENT '座位唯一ID',
    train_id      VARCHAR(20)  NOT NULL           COMMENT '所属车次编号',
    carriage_no   INT          NOT NULL           COMMENT '车厢号',
    seat_no       VARCHAR(10)  NOT NULL           COMMENT '座位号，如 01A',
    seat_bitmap   INT          DEFAULT 0          COMMENT '二进制区间占用状态：0=全部空闲',
    default_mask  INT          DEFAULT 0          COMMENT '默认掩码（每个座位固有的区间标识）',
    UNIQUE KEY uk_train_seat (train_id, carriage_no, seat_no)
) ENGINE=InnoDB COMMENT='高并发座位区间状态表（位图模型核心）';
```

### 字段说明

| 列名 | 类型 | 键 | 非空 | 默认值 | 说明 |
|------|------|:--:|:----:|--------|------|
| `seat_id` | INT | PK | ✅ | AUTO | 座位全局唯一 ID |
| `train_id` | VARCHAR(20) | UNI | ✅ | — | 所属车次编号（外键关联 `train_info`） |
| `carriage_no` | INT | UNI | ✅ | — | 车厢号（从 1 开始） |
| `seat_no` | VARCHAR(10) | UNI | ✅ | — | 座位号，如 `01A`、`02B` |
| `seat_bitmap` | INT | | | 0 | **🎯 核心字段**：二进制位图，每一位代表一个行驶区间的占用状态。0 = 空闲，1 = 已售 |
| `default_mask` | INT | | | 0 | 该座位的默认区间掩码（db_setup.py 动态添加列），用于演示时的位图初始化 |

### 唯一约束

`(train_id, carriage_no, seat_no)` 联合唯一，防止同一车次同一座位被重复录入。

### 位图编码规则

- 若列车沿途停靠 N 个站点，则有 **N−1 个行驶区间**
- `seat_bitmap` 从最低位（bit 0）到高位依次对应区间 0 → 区间 N−2
- `default_mask` 为每个座位分配唯一的"基区间"掩码，便于演示

### 示例数据（每车次 5 个座位，车厢 1）

| seat_id | train_id | carriage_no | seat_no | seat_bitmap | default_mask | 含义 |
|---------|----------|:-----------:|---------|:-----------:|:------------:|------|
| 1 | G101 | 1 | 01A | 0 | `0b00001` (1) | 区间 0 |
| 2 | G101 | 1 | 01B | 0 | `0b00010` (2) | 区间 1 |
| 3 | G101 | 1 | 02A | 0 | `0b00100` (4) | 区间 2 |
| 4 | G101 | 1 | 02B | 0 | `0b01000` (8) | 区间 3 |
| 5 | G101 | 1 | 03A | 0 | `0b10000` (16) | 区间 4 |
| 6 | G102 | 1 | 01A | 0 | `0b00001` (1) | 区间 0 |
| ... | ... | ... | ... | ... | ... | ... |

## 3. orders — 订单交易表

记录每一笔购票交易的完整信息，同时为退票触发器提供数据源。

### DDL

```sql
CREATE TABLE orders (
    order_id     VARCHAR(64)  PRIMARY KEY            COMMENT '订单流水号',
    user_id      VARCHAR(64)  NOT NULL               COMMENT '购票用户ID',
    train_id     VARCHAR(20)                         COMMENT '车次编号',
    seat_id      INT                                 COMMENT '关联 seat_status.seat_id',
    buy_mask     INT          DEFAULT 0              COMMENT '本次购买的区间掩码',
    price        DECIMAL(10,2) DEFAULT 0             COMMENT '订单金额',
    order_status VARCHAR(20)  DEFAULT 'PAID'         COMMENT '字符串状态：PAID/REFUNDED',
    status       INT          DEFAULT 1              COMMENT '数值状态：1=已支付, 2=已退票',
    create_time  TIMESTAMP    DEFAULT CURRENT_TIMESTAMP COMMENT '订单创建时间'
) ENGINE=InnoDB COMMENT='交易订单表';
```

### 字段说明

| 列名 | 类型 | 键 | 非空 | 默认值 | 说明 |
|------|------|:--:|:----:|--------|------|
| `order_id` | VARCHAR(64) | PK | ✅ | — | 订单流水号，格式 `ORD` + 10 位 HEX（如 `ORD`A3F2C8D01B） |
| `user_id` | VARCHAR(64) | | ✅ | — | 购票用户 ID |
| `train_id` | VARCHAR(20) | | | NULL | 车次编号 |
| `seat_id` | INT | | | NULL | 关联 `seat_status.seat_id` |
| `buy_mask` | INT | | | 0 | 本次购买锁定的区间掩码（用于退票时还原位图） |
| `price` | DECIMAL(10,2) | | | 0 | 订单金额 |
| `order_status` | VARCHAR(20) | | | 'PAID' | 字符串状态：`PAID`（已支付）/ `REFUNDED`（已退票） |
| `status` | INT | | | 1 | 数值状态：`1` = 已支付，`2` = 已退票 |
| `create_time` | TIMESTAMP | | | NOW | 订单创建时间 |

> **注意**：`order_status`（VARCHAR）与 `status`（INT）双写并存。**触发器**和**财务视图**仅依赖 `status`（INT）字段判断退票状态，因此退票操作务必设置 `status = 2`。

### 订单状态流转

```
创建订单 → status=1, order_status='PAID'
    │
    └── 退票 → status=2, order_status='REFUNDED'  ← 触发器自动恢复 seat_bitmap
```

## 5. 触发器：trg_orders_after_update_refund

当订单状态更新为已退票时，自动将对应座位的位图恢复。

### DDL

```sql
CREATE TRIGGER trg_orders_after_update_refund
AFTER UPDATE ON orders
FOR EACH ROW
    UPDATE seat_status
    SET seat_bitmap = seat_bitmap & (~ NEW.buy_mask)
    WHERE seat_id = NEW.seat_id
      AND NEW.status = 2
      AND (OLD.status IS NULL OR OLD.status != 2);
```

### 触发条件

| 条件 | 说明 |
|------|------|
| 时机 | `AFTER UPDATE` on `orders` |
| 条件 | `NEW.status = 2`（新状态为已退票） |
| 守卫 | `OLD.status IS NULL OR OLD.status != 2`（防止重复触发） |

### 核心逻辑

```text
seat_bitmap = seat_bitmap & (~ buy_mask)
```

- `~ buy_mask`：对购买掩码按位取反（退票区间位变为 0，其余为 1）
- `&`：将座位位图中对应退票区间的位清零
- **效果**：退票后释放区间，该区间可被再次购买

### 部署方式

通过 [apply_trigger_and_view.py](apply_trigger_and_view.py) 或 [apply_sql_via_pymysql.py](apply_sql_via_pymysql.py) 执行，也可直接在 MySQL 客户端运行 [sql_scripts/03_refund_trigger.sql](sql_scripts/03_refund_trigger.sql)。

## 6. 视图：v_daily_financial_report

按日期和车次汇总财务状况，供管理员查看日度营收。

### DDL

```sql
CREATE VIEW v_daily_financial_report AS
SELECT
    DATE(o.create_time)   AS sale_date,
    o.train_id,
    COUNT(*)              AS total_orders,
    SUM(CASE WHEN o.status = 1 THEN 1 ELSE 0 END)          AS paid_orders,
    SUM(CASE WHEN o.status = 2 THEN 1 ELSE 0 END)          AS refunded_orders,
    COALESCE(SUM(CASE WHEN o.status = 1 THEN o.price ELSE 0 END), 0) AS total_sales,
    COALESCE(SUM(CASE WHEN o.status = 2 THEN o.price ELSE 0 END), 0) AS refund_amount,
    COALESCE(SUM(CASE WHEN o.status = 1 THEN o.price ELSE 0 END), 0)
      - COALESCE(SUM(CASE WHEN o.status = 2 THEN o.price ELSE 0 END), 0) AS net_revenue
FROM orders o
WHERE o.create_time IS NOT NULL
GROUP BY DATE(o.create_time), o.train_id;
```

### 输出字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `sale_date` | DATE | 订单日期（基于 `create_time` 的日期部分） |
| `train_id` | VARCHAR | 车次编号 |
| `total_orders` | BIGINT | 当日该车次订单总数 |
| `paid_orders` | BIGINT | 已支付订单数（`status = 1`） |
| `refunded_orders` | BIGINT | 已退票订单数（`status = 2`） |
| `total_sales` | DECIMAL | 已支付订单总金额 |
| `refund_amount` | DECIMAL | 已退票订单总金额 |
| `net_revenue` | DECIMAL | **净收入** = `total_sales` − `refund_amount` |

### 使用示例

```sql
-- 查看当日汇总
SELECT * FROM v_daily_financial_report WHERE sale_date = CURDATE();

-- 查看某车次历史营收
SELECT * FROM v_daily_financial_report WHERE train_id = 'G101' ORDER BY sale_date DESC;
```

## 7. 核心设计理念：位图区间售票模型

### 问题背景

传统"一票一座"模型无法表达**同一座位在不同区间被不同乘客购买**的场景（如：北京→济南的座位，济南→上海段可再售）。

### 位图方案

```
列车经停 6 站（北京南 → 济南西 → 徐州东 → 南京南 → 苏州北 → 上海虹桥）

行驶区间（5段）：  [0]        [1]        [2]        [3]        [4]

seat_bitmap (INT, 32位):
  bit 4   bit 3   bit 2   bit 1   bit 0    ← 低位对应第 0 区间
   0        1       1       0       0      = 0b01100 = 12
                        已售出                     已售出

表示：区间 2（徐州东→南京南）和区间 3（南京南→苏州北）已被占用，
     区间 0、1、4 仍可购买。
```

### 冲突检测（按位与）

```sql
-- 请求购买 mask = 0b00100（区间2），检测冲突：
SELECT (seat_bitmap & mask) AS conflict FROM seat_status WHERE ...;

-- seat_bitmap = 0b01100, mask = 0b00100
-- 0b01100 & 0b00100 = 0b00100 ≠ 0  → 冲突！拒绝 ❌
```

### 区间占用（按位或）

```sql
-- 请求购买 mask = 0b00001（区间0），检测通过后：
UPDATE seat_status
SET seat_bitmap = seat_bitmap | mask   -- 0b01100 | 0b00001 = 0b01101
WHERE ...;
```

### 区间释放（按位与取反）

```sql
-- 退票 mask = 0b00001（区间0），触发器自动执行：
UPDATE seat_status
SET seat_bitmap = seat_bitmap & (~ mask)  -- 0b01101 & (~0b00001) = 0b01101 & 0b11110 = 0b01100
WHERE ...;
```

### 并发安全

在事务中使用 `SELECT ... FOR UPDATE` 对目标行加**排他锁（X-Lock）**，确保同一座位在同一时刻只有一个事务能读取并修改 `seat_bitmap`，从数据库层面彻底杜绝超卖。

### 优势

| 特性 | 传统方案 | 位图方案 |
|------|----------|----------|
| 区间拆分售票 | 需多表 JOIN + 复杂逻辑 | 单字段位运算 |
| 冲突检测 | O(N²) 遍历比较 | O(1) 按位与 |
| 并发安全 | 需应用层分布式锁 | 数据库行级锁即够用 |
| 可扩展性 | 区间增多 → 行数指数增长 | 1 个 INT 支持 32 区间 |
